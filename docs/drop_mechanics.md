# Механика дропа (Adventure) — справка и статистика

Документ описывает, как работает выпадение предметов после Приключений (Adventure) в 2Walks, и собирает в одном месте **вероятности дропа** по всем walk × luck. Нужен чтобы:

1. Понимать реальную картину шансов (а не угадывать по константам).
2. **Точечно настраивать баланс** — какую константу крутить и к чему это приведёт.
3. Зафиксировать дизайн **pity-системы** (задача 4.19) вместе с подтверждающей статистикой, чтобы будущий maintainer не переоткрывал анализ заново.

Источники правды в коде: [`drop.py`](../drop.py) (ролы + константы), [`adventure_data.py`](../adventure_data.py) (per-walk таблицы тиеров), [`drop_test_montecarlo.py`](../drop_test_montecarlo.py) (MC-симулятор баланса). Аналитический расчёт — `drop.compute_grade_probabilities(adv, state)`.

Status: **актуально на 0.2.6n (17.06.2026)**. Pity (раздел 7) — **спроектировано, ещё НЕ реализовано** (task 4.19).

## 1. Пайплайн одного дропа

После завершения приключения `Adventure.adventure_check_done` зовёт `Drop_Item.item_collect(hard, state)`. Он собирает предмет из независимых под-роллов:

| Под-ролл | Метод | Что определяет |
|---|---|---|
| Грейд | `one_item_random_grade(hard, state)` | `c/b/a/s/s+` или `None` (miss) — **двухстадийный**, см. §2 |
| Тип | `item_type(state)` | ring / necklace / helmet / shoes / t-shirt / backpack — взвешенно, см. §5 |
| Характеристика | `characteristic_type(state)` | stamina / energy_max / speed_skill / luck (для рюкзака — `backpack_capacity`) |
| Бонус | `item_bonus_value(grade)` | +1..+5 по грейду (c→1 … s+→5) |
| Качество | `item_quality(state)` | `randint(20 + luck, 100)` |
| Цена | `item_price(grade, quality)` | `quality × {c:0.5, b:1, a:1.5, s:2, s+:2.5}` |

Если грейд = `None` — дропа нет («Ничего не выпало»). Иначе предмет кладётся в инвентарь / `pending_drop` / форс-продаётся (3 ветки по заполненности рюкзака, см. CLAUDE.md / task 4.50.1).

**`current_luck(state)` = `gym.luck_skill + equipment_luck_bonus + char_level.skill_luck`** — вычисляется в момент дропа, влияет почти на все под-роллы.

## 2. Грейд: двухстадийный ролл (ядро шанса дропа)

Это самое важное для баланса. «Выпадет ли что-то и какого грейда» решается в **две стадии**:

### Стадия 1 — гейт (общий «выпадет ли вообще»)

```python
i = randint(1, 100 - luck)
if i <= drop_percent_gl:   # drop_percent_gl = 80
    ...                    # проходим к стадии 2
```

`P(gate) = min(drop_percent_gl, 100−luck) / (100−luck)`. При luck=0 → 80%. **Важно:** растёт с luck, потому что сужается диапазон `randint`. При `luck ≥ 100 − drop_percent_gl` (т.е. luck ≥ 20) гейт всегда проходит.

### Стадия 2 — конкуренция тиеров грейда

Для walk бросается по одному роллу на каждый доступный тир (`randint(1, 100−luck)`); побеждает тир с наименьшим роллом, который при этом ≤ своего порога. Пример walk_hard (3 тира c/b/a):

```python
c = randint(1, 100 - luck); b = randint(1, 100 - luck); a = randint(1, 100 - luck)
if   c < b and c < a and c <= 75: return 'c-grade'   # drop_percent_item_c
elif b < c and b < a and b <= 60: return 'b-grade'   # drop_percent_item_b
elif a < c and a < b and a <= 45: return 'a-grade'   # drop_percent_item_a
```

Аналитика обеих стадий — `compute_grade_probabilities` (формула в docstring `drop.py`). MC-parity тесты в `tests/test_drop.py` сверяют аналитику с реальной `one_item_random_grade`.

### ⚠️ «Пол» вероятности дропа

Поскольку стадия 2 всегда режет независимо от гейта, **у P(дроп) есть нижняя граница, которую буст гейта пробить не может**. Даже при `drop_percent_gl=100` walk_30k упирается в ~35% (luck=0), walk_25k — в ~43%. Это ключевой факт, определивший выбор pity-механики (§7): **«+5 к гейту за промах» НЕ работает на endgame**, потому что упирается в этот пол. Re-roll (свежая попытка целиком) — работает, т.к. перебрасывает обе стадии.

## 5. Тип предмета + drop-modifier по luck (task 4.65, 0.2.6l)

`item_type(state)` — **взвешенная выборка** (`random.choices`). База: рюкзак 5, остальные 5 типов по 19 (сумма 100). Поверх базы — **drop-modifier**: буст шанса к «нужным» типам (где у игрока пусто/низкий грейд), пропорционально `current_luck`.

- «Нужность» типа = дистанция до S+ по грейду (`gap = 5 − лучший_имеющийся_тир`).
- Кольца (вариант R2): gap по **худшему** из двух надетых пальцев — стимул заполнить оба finger-слота.
- Относительная модель: худший слот (`max_gap`) получает полный `luck`-буст, остальные — `luck × gap / max_gap`. Всё S+ (max_gap=0) → буста нет.

Хелперы: `_item_grade_tier`, `_best_owned_tier`, `_type_gap`, `_type_weights`. Подробности — CLAUDE.md (раздел про drop.py) и TASKS 4.65.

## 6. Качество, характеристика, цена

- **Характеристика** (`characteristic_type`): 4 ролла `randint(1, 100+luck)` (stamina/energy_max/speed_skill/luck), побеждает максимум; ничья → `None`. Рюкзак — фиксированно `backpack_capacity`.
- **Качество** (`item_quality`): `randint(20 + current_luck, 100)`. luck поднимает нижнюю границу → меньше «мусорных» предметов.
- **Цена** (`item_price`): `quality × множитель грейда` (c 0.5 … s+ 2.5).

## 7. Pity-система (task 4.19) — дизайн + статистика

**Status: спроектировано, НЕ реализовано.** Финализировано в ходе анализа 17.06.2026.

### Проблема

См. §2 «пол» и §4: на endgame walk много пустых заходов и длинные серии промахов (до 27–31 подряд по MC). Хочется «bad-luck protection».

### Отвергнутая модель: «+5 к гейту за промах»

Per-walk счётчик промахов, каждый промах `drop_percent_gl += 5` (cap 100). **Не работает**, потому что бустит только стадию 1 и упирается в пол стадии 2 (§2). MC (luck=10): walk_30k 34→37% максимум, серии до 27 остаются. Saturation наступает уже на pity≈2–4 (гейт добивает до 100). Симулятор: см. `scratch_pity_sim.py:sequential_dynamics`.

### Принятая модель: RE-ROLL (вариант 2), без cap

> Per-walk счётчик `c`. Заход **полностью пустой** (все попытки — miss) → `c += 1`. Любой дроп → `c = 0`. На заходе делаем `1 + c` независимых вызовов `one_item_random_grade`, останавливаемся на первом дропе.

Почему работает: каждая лишняя попытка — свежий двухстадийный ролл, **пробивает пол** (`P = 1 − (1−p)^(1+c) → 1`). И **сохраняет grade-иерархию**: re-roll на walk_30k всегда только S+, walk_easy от re-roll'ов S+ не родит (в отличие от «бустить пороги грейда»).

### Cap

**Не нужен.** Серии и так не выходят за ~8; `cap=10` в MC давал идентичные числа. Решено не вводить ради простоты логики.

### План реализации (когда дойдём)

- `state.py`: 7 счётчиков (per-walk) + round-trip в 3 форматах (state.json / Sheets JSON-blob / legacy CSV) — типичная ловушка, см. CLAUDE.md.
- `drop.py` / `adventure.py`: цикл `1 + c` попыток в финализаторе; инкремент при полностью пустом заходе, reset при дропе. STALE-совместимость финализатора (см. `adventure_check_done` + `emit_or_defer`).
- Тесты: счётчик инкрементится/сбрасывается; цикл попыток; round-trip полей.

## 8. Как настраивать баланс (cookbook)

| Хочу | Кручу |
|---|---|
| Больше/меньше дропа в целом | `drop_percent_gl` (гейт). Но помни про «пол» §2 — на endgame эффект ограничен. |
| Сдвинуть редкость конкретного грейда | `drop_percent_item_<grade>` (или per-walk override `_walk_25k/_walk_30k`). |
| Позиционирование endgame walk | per-walk S+ override (как 0.2.4g: walk_30k=35 vs walk_25k=20). |
| Чаще нужные **типы** предметов | модель `_type_weights` / база `_BASE_TYPE_WEIGHTS` (§5, task 4.65). |
| Защита от серий невезения | pity re-roll (§7, task 4.19). |
| Меньше «мусорного» качества | нижняя граница в `item_quality` (`20 + luck`). |

**Инструменты проверки:**
- Аналитика per-grade — `drop.compute_grade_probabilities(adv, state)` (без рандома, мгновенно).
- MC баланса — `python drop_test_montecarlo.py` (использует реальную `one_item_random_grade`).
- Анализ pity (re-roll vs gate-boost, серии, стоимость) — `scratch_pity_sim.py` (временный, не в git; пересоздать при необходимости из этого дока).

После любой правки констант — прогнать MC-parity тесты `tests/test_drop.py`, чтобы аналитика и реальный ролл не разъехались.
